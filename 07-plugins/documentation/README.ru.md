# Плагин документации

Комплексная генерация и поддержка документации для вашей кодовой базы.

## Возможности

- Генерация API-документации из исходного кода
- Создание и обновление README
- Синхронизация документации с изменениями кода
- Улучшение комментариев кода
- Генерация примеров и руководств
- Проверка полноты документации

## Установка

```json
{
  "plugins": ["github:your-org/documentation-plugin"]
}
```

## Включённые команды

- `/generate-api-docs` — Создать API-документацию из исходного кода
- `/generate-readme` — Создать или обновить README проекта
- `/sync-docs` — Синхронизировать документацию с изменениями кода
- `/validate-docs` — Проверить документацию на полноту и точность

## Включённые субагенты

- **api-documenter** — Специалист по API-документации
- **code-commentator** — Специалист по встроенной документации
- **example-generator** — Специалист по примерам кода

## Включённые шаблоны

- `templates/api-endpoint.md` — Шаблон документации эндпоинта
- `templates/function-docs.md` — Шаблон документации функции
- `templates/adr-template.md` — Шаблон записи архитектурных решений

## Включённые MCP-серверы

- **filesystem** — Доступ к файлам документации

## Использование

```bash
# Создать API-документацию
/generate-api-docs

# Создать README
/generate-readme

# Синхронизировать документацию после изменений кода
/sync-docs

# Проверить документацию
/validate-docs
```

## Требования

- Исходный код с JSDoc/docstring-комментариями (рекомендуется)
- Права на запись в директорию документации

## Пример рабочего процесса

1. `/generate-api-docs` — Создать первоначальную API-документацию
2. `/generate-readme` — Создать README с обзором проекта
3. После изменений кода: `/sync-docs`
4. Перед релизом: `/validate-docs`

## Лучшие практики

- Запускать `/sync-docs` после каждого крупного PR
- Использовать шаблоны для единообразного форматирования
- Проверять документацию перед каждым релизом
- Хранить примеры вместе с документацией
