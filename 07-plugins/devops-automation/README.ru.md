# Плагин DevOps-автоматизации

Комплексная DevOps-автоматизация для деплоя, мониторинга и реагирования на инциденты.

## Возможности

- Полная автоматизация деплоя (blue-green, canary)
- Мониторинг и анализ оповещений
- Реагирование на инциденты и управление ими
- Автоматические откаты при сбоях
- Интеграция со Slack для уведомлений
- Интеграция с Kubernetes и облачными провайдерами

## Установка

```json
{
  "plugins": ["github:your-org/devops-automation-plugin"]
}
```

## Включённые команды

- `/deploy` — Задеплоить приложение в production или staging
- `/rollback` — Откатить к предыдущему деплою
- `/incident` — Запустить реагирование на инцидент
- `/status` — Проверить состояние системы

## Включённые субагенты

- **alert-analyzer** — Анализирует мониторинговые оповещения
- **deployment-specialist** — Управляет операциями деплоя
- **incident-commander** — Координирует реагирование на инциденты

## Включённые MCP-серверы

- **kubernetes** — Управление кластером Kubernetes
- **prometheus** — Запросы метрик и оповещений
- **pagerduty** — Управление инцидентами

## Скрипты

- `scripts/deploy.sh` — Скрипт деплоя
- `scripts/rollback.sh` — Скрипт отката
- `scripts/health-check.sh` — Скрипт проверки работоспособности

## Хуки

- **PostDeploy** — Запускает проверки работоспособности после деплоя
- **OnAlert** — Инициирует анализ при критических оповещениях

## Использование

```bash
# Задеплоить в production
/deploy production

# Проверить статус всех сервисов
/status

# Начать реагирование на инцидент
/incident severity=high

# Откатить при необходимости
/rollback
```

## Требования

- kubectl настроен и подключён
- Prometheus/AlertManager доступен
- PagerDuty API-ключ (опционально)
- Доступ к Slack Webhook

## Конфигурация

```json
{
  "devops": {
    "kubernetes_context": "production",
    "slack_webhook": "${SLACK_WEBHOOK_URL}",
    "pagerduty_key": "${PAGERDUTY_API_KEY}",
    "health_check_timeout": 300
  }
}
```

## Пример рабочего процесса

1. `/deploy production` — деплой запущен
2. Автоматические проверки работоспособности
3. При сбое: `/rollback` с автоматическим уведомлением
4. При оповещении мониторинга: `/incident` для структурированного реагирования
5. Post-mortem и документация
