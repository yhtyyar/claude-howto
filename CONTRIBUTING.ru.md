# Внесение вклада в Claude How To

Благодарим за интерес к внесению вклада в этот проект! Это руководство поможет вам понять, как эффективно вносить вклад.

## О проекте

Claude How To — это визуальное руководство на основе примеров для Claude Code. Мы предоставляем:
- **Диаграммы Mermaid** с объяснением работы функций
- **Готовые к использованию шаблоны** производственного уровня
- **Реальные примеры** с контекстом и лучшими практиками
- **Прогрессивные пути обучения** от начинающего до продвинутого

## Типы вкладов

### 1. Новые примеры или шаблоны
Добавляйте примеры для существующих функций (слеш-команды, навыки, хуки и т.д.):

- Чёткие объяснения принципов работы
- Сценарии использования и преимущества
- Советы по устранению неполадок

### 2. Улучшение документации
- Разъяснение запутанных разделов
- Исправление опечаток и грамматических ошибок
- Добавление недостающей информации
- Улучшение примеров кода

### 3. Руководства по функциям
Создавайте руководства для новых функций Claude Code:
- Пошаговые руководства
- Архитектурные диаграммы
- Типичные паттерны и антипаттерны
- Реальные рабочие процессы

### 4. Отчёты об ошибках
Сообщайте о проблемах, с которыми столкнулись:
- Опишите, что ожидали
- Опишите, что произошло на самом деле
- Включите шаги для воспроизведения
- Добавьте версию Claude Code и ОС

### 5. Отзывы и предложения
Помогайте улучшать руководство:
- Предлагайте лучшие объяснения
- Указывайте пробелы в охвате
- Рекомендуйте новые разделы или реорганизацию

## Начало работы

### 1. Форк и клонирование
```bash
git clone https://github.com/luongnv89/claude-howto.git
cd claude-howto
```

### 2. Создание ветки
Используйте описательное имя ветки:
```bash
git checkout -b add/feature-name
git checkout -b fix/issue-description
git checkout -b docs/improvement-area
```

### 3. Настройка окружения
```bash
# Создать виртуальное окружение
python3 -m venv .venv
source .venv/bin/activate

# Установить pre-commit хуки (опционально, но рекомендуется)
pip install pre-commit
pre-commit install

# Запустить pre-commit проверки вручную
pre-commit run --all-files
```

## Как вносить примеры

### Добавление слеш-команды
1. Создайте `.md` файл в `01-slash-commands/`
2. Включите:
   - Чёткое описание функциональности
   - Сценарии использования
   - Инструкции по установке
   - Примеры использования
   - Советы по кастомизации
3. Обновите `01-slash-commands/README.md`

### Добавление навыка
1. Создайте директорию в `03-skills/`
2. Включите:
   - `SKILL.md` — Основная документация
   - `scripts/` — Вспомогательные скрипты при необходимости
   - `templates/` — Шаблоны промптов
   - Пример использования в README
3. Обновите `03-skills/README.md`

### Добавление субагента
1. Создайте `.md` файл в `04-subagents/`
2. Включите:
   - Назначение и возможности агента
   - Структуру системного промпта
   - Примеры сценариев использования
   - Примеры интеграции
3. Обновите `04-subagents/README.md`

### Добавление MCP-конфигурации
1. Создайте `.json` файл в `05-mcp/`
2. Включите:
   - Объяснение конфигурации
   - Требуемые переменные окружения
   - Инструкции по настройке
   - Примеры использования
3. Обновите `05-mcp/README.md`

### Добавление хука
1. Создайте `.sh` файл в `06-hooks/`
2. Включите:
   - Shebang и описание
   - Понятные комментарии, объясняющие логику
   - Обработку ошибок
   - Вопросы безопасности
3. Обновите `06-hooks/README.md`

## Руководства по написанию

### Стиль Markdown
- Используйте чёткие заголовки (H2 для разделов, H3 для подразделов)
- Держите абзацы короткими и сфокусированными
- Используйте маркированные списки
- Включайте блоки кода с указанием языка
- Добавляйте пустые строки между разделами

### Примеры кода
- Делайте примеры готовыми к копированию
- Комментируйте неочевидную логику
- Включайте как простые, так и продвинутые версии
- Показывайте реальные сценарии использования
- Выделяйте потенциальные проблемы

### Документация
- Объясняйте «почему», а не только «что"
- Включайте предварительные требования
- Добавляйте разделы по устранению неполадок
- Ссылайтесь на связанные темы
- Держите доступным для начинающих

### JSON/YAML
- Используйте правильные отступы (2 или 4 пробела последовательно)
- Добавляйте комментарии, объясняющие конфигурацию
- Включайте примеры валидации

### Диаграммы
- Используйте Mermaid, когда возможно
- Держите диаграммы простыми и читаемыми
- Включайте описания под диаграммами
- Ссылайтесь на соответствующие разделы

## Руководство по коммитам

Следуйте формату conventional commit:
```
type(scope): description

[optional body]
```

Типы:
- `feat`: Новая функция или пример
- `fix`: Исправление ошибки
- `docs`: Изменения документации
- `refactor`: Реструктуризация кода
- `style`: Изменения форматирования
- `test`: Добавление или изменение тестов
- `chore`: Сборка, зависимости и т.д.

Примеры:
```
feat(slash-commands): Add API documentation generator
docs(memory): Improve personal preferences example
fix(README): Correct table of contents link
docs(skills): Add comprehensive code review skill
```

## Процесс Pull Request

1. **Создайте PR с чётким описанием**:
   - Что добавляет/исправляет?
   - Зачем это нужно?
   - Связанные задачи (если есть)

2. **Включите соответствующие детали**:
   - Новая функция? Включите сценарии использования
   - Документация? Объясните улучшения
   - Примеры? Покажите до/после

3. **Свяжите с задачами**:
   - Используйте `Closes #123` для автоматического закрытия связанных задач

4. **Будьте терпеливы при ревью**:
   - Мейнтейнеры могут предлагать улучшения
   - Итерируйтесь на основе обратной связи
   - Окончательное решение принимают мейнтейнеры

## Процесс ревью кода

Ревьюверы проверят:
- **Точность**: Работает ли как описано?
- **Качество**: Готово ли к использованию в production?
- **Консистентность**: Следует ли паттернам проекта?
- **Документация**: Ясно ли и полно?
- **Безопасность**: Есть ли уязвимости?

## Политики проекта

### Конфиденциальная информация
- Никогда не коммитьте API-ключи, токены или учётные данные
- Используйте плейсхолдеры в примерах
- Включайте `.env.example` для конфигурационных файлов
- Документируйте требуемые переменные окружения

### Качество кода

- Комментируйте неочевидную логику

### Интеллектуальная собственность

- Уважайте существующие авторские права
