## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ОБЯЗАНЫ** учитывать ввод пользователя перед началом работы (если не пустой). Пользователь может указать:
- `create` — создать новый CLAUDE.md с нуля
- `update` — улучшить существующий CLAUDE.md
- `audit` — проанализировать и отчитаться о качестве текущего CLAUDE.md
- Конкретный путь для создания/обновления (например, `src/api/CLAUDE.md` для инструкций конкретной директории)

## Основные принципы

**LLM не имеют состояния**: CLAUDE.md — единственный файл, автоматически включаемый в каждый разговор. Он служит основным документом онбординга AI-агентов в вашу кодовую базу.

### Золотые правила

1. **Меньше — значит больше**: Frontier LLM могут следовать ~150-200 инструкциям. Системный промпт Claude Code уже использует ~50. Держите CLAUDE.md сфокусированным и кратким.

2. **Универсальная применимость**: Включайте только информацию, актуальную для КАЖДОЙ сессии. Специфические для задач инструкции принадлежат отдельным файлам.

3. **Не используйте Claude как линтер**: Руководства по стилю раздувают контекст и ухудшают следование инструкциям. Используйте детерминированные инструменты (prettier, eslint и т.д.).

4. **Никогда не автогенерировать**: CLAUDE.md — самая высокоэффективная точка AI-упряжи. Создавайте его вручную с тщательным обдумыванием.

## Поток выполнения

### 1. Анализ проекта

Сначала проанализируйте текущее состояние проекта:

1. Проверить существующие файлы CLAUDE.md:
   - Корневой уровень: `./CLAUDE.md` или `.claude/CLAUDE.md`
   - Специфичные для директорий: `**/CLAUDE.md`
   - Глобальная конфигурация пользователя: `~/.claude/CLAUDE.md`

2. Определить структуру проекта:
   - Технический стек (языки, фреймворки)
   - Тип проекта (монорепо, одиночное приложение, библиотека)
   - Инструменты разработки (менеджер пакетов, система сборки, тест-раннер)

3. Просмотреть существующую документацию:
   - README.md
   - CONTRIBUTING.md
   - package.json, pyproject.toml, Cargo.toml и т.д.

### 2. Стратегия содержимого (ЧТО, ПОЧЕМУ, КАК)

Структурируйте CLAUDE.md вокруг трёх измерений:

#### ЧТО — Технология и структура
- Обзор технического стека
- Организация проекта (особенно важно для монорепо)
- Ключевые директории и их назначения

#### ПОЧЕМУ — Цель и контекст
- Что делает проект
- Почему были приняты определённые архитектурные решения
- За что отвечает каждый основной компонент

#### КАК — Рабочий процесс и соглашения
- Рабочий процесс разработки (bun vs node, pip vs uv и т.д.)
- Процедуры и команды тестирования
- Методы верификации и сборки
- Критические «подводные камни» или неочевидные требования

### 3. Стратегия постепенного раскрытия

Для крупных проектов рекомендовать создание папки `agent_docs/`:

```
agent_docs/
  |- building_the_project.md
  |- running_tests.md
  |- code_conventions.md
  |- architecture_decisions.md
```

В CLAUDE.md ссылаться на эти файлы с инструкциями типа:
```markdown
Подробные инструкции по сборке см. в `agent_docs/building_the_project.md`
```

**Важно**: Используйте ссылки `файл:строка` вместо фрагментов кода для предотвращения устаревшего контекста.

### 4. Ограничения качества

При создании или обновлении CLAUDE.md:

1. **Целевая длина**: Менее 300 строк (в идеале менее 100)
2. **Без правил стиля**: Удалить инструкции по линтингу/форматированию
3. **Без специфических для задач инструкций**: Перенести в отдельные файлы
4. **Без фрагментов кода**: Использовать ссылки на файлы
5. **Без избыточной информации**: Не повторять то, что есть в package.json или README

### 5. Необходимые разделы

Хорошо структурированный CLAUDE.md должен включать:

```markdown
# Название проекта

Краткое однострочное описание.

## Технический стек
- Основной язык и версия
- Ключевые фреймворки/библиотеки
- База данных/хранилище (если есть)

## Структура проекта
[Только для монорепо или сложных структур]
- `apps/` — Точки входа приложений
- `packages/` — Общие библиотеки

## Команды разработки
- Установка: `команда`
- Тесты: `команда`
- Сборка: `команда`

## Критические соглашения
[Только неочевидные, высокоэффективные соглашения]
- Соглашение 1 с кратким объяснением
- Соглашение 2 с кратким объяснением

## Известные проблемы / Подводные камни
[Вещи, которые регулярно сбивают разработчиков]
- Проблема 1
- Проблема 2
```

### 6. Антипаттерны для избегания

**НЕ включайте:**
- Руководства по стилю кода (используйте линтеры)
- Документацию по использованию Claude
- Длинные объяснения очевидных паттернов
- Скопированные примеры кода
- Общие лучшие практики («пишите чистый код»)
- Инструкции для конкретных задач
- Автогенерированный контент
- Обширные списки задач

### 7. Чек-лист валидации

Перед финализацией убедиться:

- [ ] Менее 300 строк (предпочтительно менее 100)
- [ ] Каждая строка применима ко ВСЕМ сессиям
- [ ] Нет правил стиля/форматирования
- [ ] Нет фрагментов кода (используйте ссылки на файлы)
- [ ] Команды проверены и работают
- [ ] Постепенное раскрытие используется для сложных проектов
- [ ] Критические подводные камни задокументированы
- [ ] Нет избыточности с README.md

## Формат вывода

### Для `create` или по умолчанию:

1. Проанализировать проект
2. Написать CLAUDE.md следуя структуре выше
3. Представить черновик для ревью
4. Записать в соответствующее место после одобрения

### Для `update`:

1. Прочитать существующий CLAUDE.md
2. Провести аудит по лучшим практикам
3. Определить:
   - Контент для удаления (правила стиля, фрагменты кода, специфические для задач)
   - Контент для сжатия
   - Недостающую важную информацию
4. Представить изменения для ревью
5. Применить изменения после одобрения

### Для `audit`:

1. Прочитать существующий CLAUDE.md
2. Сформировать отчёт с:
   - Текущим количеством строк vs. целевое
   - Процентом универсально применимого контента
   - Списком найденных антипаттернов
   - Рекомендациями по улучшению
3. НЕ изменять файл, только отчитываться

## Обработка AGENTS.md

При запросе создания/обновления AGENTS.md:

AGENTS.md используется для определения специализированных поведений агентов. В отличие от CLAUDE.md (контекст проекта), AGENTS.md определяет:
- Пользовательские роли и возможности агентов
- Специфические для агентов инструкции и ограничения
- Определения рабочих процессов для сценариев с несколькими агентами

Применять схожие принципы:
- Сохранять сфокусированность и краткость
- Использовать постепенное раскрытие
- Ссылаться на внешние документы вместо встраивания контента

## Примечания

- Всегда проверять, что команды работают, перед их включением
- При сомнении — не включать (меньше — значит больше)
- Системное напоминание сообщает Claude, что CLAUDE.md «может или не может быть актуальным» — чем больше шума, тем больше его игнорируют
- Монорепо больше всего выигрывают от чёткой структуры ЧТО/ПОЧЕМУ/КАК
- Файлы CLAUDE.md для конкретных директорий должны быть ещё более сфокусированными
