# Каталог запахов кода

Подробный справочник по запахам кода на основе книги Мартина Фаулера *«Рефакторинг»* (2-е издание). Запахи кода — это симптомы более глубоких проблем, они указывают на то, что с дизайном кода что-то не так.

> «Запах кода — это поверхностный признак, который обычно соответствует более глубокой проблеме в системе.» — Мартин Фаулер

---

## Разбухшие элементы

Запахи кода, представляющие нечто, разросшееся настолько, что им трудно управлять.

### Длинный метод

**Признаки:**
- Метод превышает 30-50 строк
- Нужно прокручивать, чтобы увидеть весь метод
- Несколько уровней вложенности
- Комментарии, объясняющие что делают разделы

**Почему плохо:**
- Трудно понять
- Трудно тестировать изолированно
- Изменения имеют непредвиденные последствия
- Дублирующаяся логика скрывается внутри

**Рефакторинги:**
- Извлечение метода
- Замена временной переменной запросом
- Введение объекта-параметра
- Замена метода объектом метода
- Декомпозиция условия

**Пример (До):**
```javascript
function processOrder(order) {
  // Валидация заказа (20 строк)
  if (!order.items) throw new Error('Нет товаров');
  if (order.items.length === 0) throw new Error('Пустой заказ');
  // ... ещё валидация

  // Вычисление итогов (30 строк)
  let subtotal = 0;
  for (const item of order.items) {
    subtotal += item.price * item.quantity;
  }
  // ... налог, доставка, скидки

  // Отправка уведомлений (20 строк)
  // ... логика email
}
```

**Пример (После):**
```javascript
function processOrder(order) {
  validateOrder(order);
  const totals = calculateOrderTotals(order);
  sendOrderNotifications(order, totals);
  return { order, totals };
}
```

### Большой класс

**Признаки:**
- Класс имеет много переменных экземпляра (>7-10)
- Класс имеет много методов (>15-20)
- Имя класса расплывчато (Manager, Handler, Processor)
- Методы не используют все переменные экземпляра

**Почему плохо:**
- Нарушает принцип единственной ответственности
- Трудно тестировать
- Изменения распространяются на несвязанные функции
- Трудно переиспользовать части

**Рефакторинги:**
- Извлечение класса
- Извлечение подкласса
- Извлечение интерфейса

**Обнаружение:**
```
Строк кода > 300
Количество методов > 15
Количество полей > 10
```

---

### Примитивная одержимость

**Признаки:**
- Использование примитивов для доменных концепций (string для email, int для денег)
- Массивы примитивов вместо объектов
- Строковые константы для кодов типов
- Магические числа/строки

**Почему плохо:**
- Нет валидации на уровне типа
- Логика рассыпана по всей кодовой базе
- Легко передать неверные значения
- Отсутствующие доменные концепции

**Рефакторинги:**
- Замена примитива объектом
- Замена кода типа классом
- Замена кода типа подклассами
- Замена кода типа состоянием/стратегией

**Пример (До):**
```javascript
const user = {
  email: 'john@example.com',     // Просто строка
  phone: '1234567890',           // Просто строка
  status: 'active',              // Магическая строка
  balance: 10050                 // Центы как целое число
};
```

**Пример (После):**
```javascript
const user = {
  email: new Email('john@example.com'),
  phone: new PhoneNumber('1234567890'),
  status: UserStatus.ACTIVE,
  balance: Money.cents(10050)
};
```

### Длинный список параметров

**Признаки:**
- Методы с 4+ параметрами
- Параметры, которые всегда появляются вместе
- Булевы флаги, изменяющие поведение метода
- Часто передаётся null/undefined

**Почему плохо:**
- Трудно вызвать правильно
- Путаница с порядком параметров
- Указывает, что метод делает слишком много
- Трудно добавлять новые параметры

**Рефакторинги:**
- Введение объекта-параметра
- Сохранение целого объекта
- Замена параметра вызовом метода
- Удаление флагового аргумента

**Пример (До):**
```javascript
function createUser(firstName, lastName, email, phone,
                    street, city, state, zip,
                    isAdmin, isActive, createdBy) {
  // ...
}
```

**Пример (После):**
```javascript
function createUser(personalInfo, address, options) {
  // personalInfo: { firstName, lastName, email, phone }
  // address: { street, city, state, zip }
  // options: { isAdmin, isActive, createdBy }
}
```

---

### Группы данных

**Признаки:**
- Одни и те же 3+ поля появляются вместе многократно
- Параметры, которые всегда путешествуют вместе
- Классы с подмножествами полей, принадлежащих вместе

**Почему плохо:**
- Дублирующаяся логика обработки
- Отсутствующая абстракция
- Труднее расширять
- Указывает на скрытый класс

**Рефакторинги:**
- Извлечение класса
- Введение объекта-параметра
- Сохранение целого объекта

**Пример:**
```javascript
// Группа данных: координаты (x, y, z)
function movePoint(x, y, z, dx, dy, dz) { }
function scalePoint(x, y, z, factor) { }
function distanceBetween(x1, y1, z1, x2, y2, z2) { }

// Извлечь класс Point3D
class Point3D {
  constructor(x, y, z) { }
  move(delta) { }
  scale(factor) { }
  distanceTo(other) { }
}
```

## Нарушители объектно-ориентированного подхода

Запахи, указывающие на неполное или неправильное использование принципов ООП.

### Switch-операторы

**Признаки:**
- Длинные switch/case или if/else цепочки
- Один и тот же switch в нескольких местах
- Switch по кодам типов
- Добавление новых случаев требует изменений везде

**Почему плохо:**
- Нарушает принцип открытости/закрытости
- Изменения распространяются на все места с switch
- Трудно расширять
- Часто указывает на отсутствующий полиморфизм

**Рефакторинги:**
- Замена условного полиморфизмом
- Замена кода типа подклассами
- Замена кода типа состоянием/стратегией

**Пример (До):**
```javascript
function calculatePay(employee) {
  switch (employee.type) {
    case 'hourly':
      return employee.hours * employee.rate;
    case 'salaried':
      return employee.salary / 12;
    case 'commissioned':
      return employee.sales * employee.commission;
  }
}
```

**Пример (После):**
```javascript
class HourlyEmployee {
  calculatePay() {
    return this.hours * this.rate;
  }
}

class SalariedEmployee {
  calculatePay() {
    return this.salary / 12;
  }
}
```

---

### Временное поле

**Признаки:**
- Переменные экземпляра, используемые только в некоторых методах
- Поля, задаваемые условно
- Сложная инициализация для определённых случаев

**Почему плохо:**
- Сбивает с толку — поле существует, но может быть null
- Трудно понять состояние объекта
- Указывает на скрытую условную логику

**Рефакторинги:**
- Извлечение класса
- Введение объекта Null
- Замена временного поля локальной переменной

---

### Отказ от наследства

**Признаки:**
- Подкласс не использует унаследованные методы/данные
- Подкласс переопределяет, чтобы ничего не делать
- Наследование используется для повторного использования кода, а не для отношения IS-A

**Почему плохо:**
- Неправильная абстракция
- Нарушает принцип подстановки Лисков
- Вводящая в заблуждение иерархия

**Рефакторинги:**
- Снижение метода/поля
- Замена подкласса делегатом
- Замена наследования делегированием

---

### Альтернативные классы с разными интерфейсами

**Признаки:**
- Два класса, делающих похожие вещи
- Разные имена методов для одной концепции
- Могут использоваться взаимозаменяемо

**Почему плохо:**
- Дублирующиеся реализации
- Нет общего интерфейса
- Трудно переключаться между ними

**Рефакторинги:**
- Переименование метода
- Перенос метода
- Извлечение суперкласса
- Извлечение интерфейса

---

## Препятствующие изменениям

Запахи, которые затрудняют изменения — изменение одного требует изменения многого другого.

### Дивергентное изменение

**Признаки:**
- Один класс изменяется по нескольким разным причинам
- Изменения в разных областях вызывают правки одного класса
- Класс является «Богом-классом»

**Почему плохо:**
- Нарушает принцип единственной ответственности
- Высокая частота изменений
- Конфликты слияния

**Рефакторинги:**
- Извлечение класса
- Извлечение суперкласса
- Извлечение подкласса

**Пример:**
Класс `User` изменяется из-за:
- Изменений аутентификации
- Изменений профиля
- Изменений выставления счетов
- Изменений уведомлений

→ Извлечь: `AuthService`, `ProfileService`, `BillingService`, `NotificationService`

---

### Операция дробовика

**Признаки:**
- Одно изменение требует правок во многих классах
- Небольшая функция требует касания 10+ файлов
- Изменения рассыпаны, трудно найти все

**Почему плохо:**
- Легко пропустить место
- Высокая связность
- Изменения чреваты ошибками

**Рефакторинги:**
- Перенос метода
- Перенос поля
- Встраивание класса

**Обнаружение:**
Искать: добавление одного поля требует изменений в >5 файлах.

---

### Параллельные иерархии наследования

**Признаки:**
- Создание подкласса в одной иерархии требует подкласса в другой
- Префиксы классов совпадают (например, `DatabaseOrder`, `DatabaseProduct`)

**Почему плохо:**
- Двойное обслуживание
- Связность между иерархиями
- Легко забыть одну сторону

**Рефакторинги:**
- Перенос метода
- Перенос поля
- Устранение одной иерархии

---

## Лишнее

Нечто ненужное, что следует удалить.

### Комментарии (избыточные)

**Признаки:**
- Комментарии, объясняющие что делает код
- Закомментированный код
- TODO/FIXME, которые висят вечно
- Извинения в комментариях

**Почему плохо:**
- Комментарии лгут (выходят из синхронизации)
- Код должен быть самодокументирующимся
- Мёртвый код вызывает путаницу

**Рефакторинги:**
- Извлечение метода (имя объясняет что)
- Переименование (ясность без комментариев)
- Удаление закомментированного кода
- Введение утверждения

**Хорошие vs плохие комментарии:**
```javascript
// ПЛОХО: Объяснение что
// Перебрать пользователей и проверить, активны ли они
for (const user of users) {
  if (user.status === 'active') { }
}

// ХОРОШО: Объяснение почему
// Только активные пользователи — неактивные обрабатываются задачей очистки
const activeUsers = users.filter(u => u.isActive);
```

---

### Дублированный код

**Признаки:**
- Одинаковый код в нескольких местах
- Похожий код с небольшими вариациями
- Паттерны копи-паста

**Почему плохо:**
- Исправления багов нужно делать в нескольких местах
- Риск несогласованности
- Раздутая кодовая база

**Рефакторинги:**
- Извлечение метода
- Извлечение класса
- Подъём метода (в иерархиях)
- Шаблонный метод

**Правило обнаружения:**
Любой код, продублированный 3+ раза, следует извлечь.

---

### Ленивый класс

**Признаки:**
- Класс не делает достаточно, чтобы оправдать своё существование
- Обёртка без добавленной ценности
- Результат чрезмерного проектирования

**Почему плохо:**
- Накладные расходы на обслуживание
- Ненужная косвенность
- Сложность без выгоды

**Рефакторинги:**
- Встраивание класса
- Свёртывание иерархии

---

### Мёртвый код

**Признаки:**
- Недостижимый код
- Неиспользуемые переменные/методы/классы
- Закомментированный код
- Код за невозможными условиями

**Почему плохо:**
- Путаница
- Бремя обслуживания
- Замедляет понимание

**Рефакторинги:**
- Удаление мёртвого кода
- Безопасное удаление

**Обнаружение:**
```bash
# Искать неиспользуемые экспорты
# Искать несвязанные функции
# Предупреждения IDE об «unused»
```

---

### Спекулятивная обобщённость

**Признаки:**
- Абстрактные классы с одним подклассом
- Неиспользуемые параметры «на будущее»
- Методы, которые только делегируют
- «Фреймворк» для одного случая использования

**Почему плохо:**
- Сложность без выгоды
- YAGNI (You Ain't Gonna Need It — Вам это не понадобится)
- Труднее понять

**Рефакторинги:**
- Свёртывание иерархии
- Встраивание класса
- Удаление параметра
- Переименование метода

---

## Чрезмерная связность

Запахи, представляющие чрезмерную связность между классами.

### Зависть к функциям

**Признаки:**
- Метод использует больше данных другого класса, чем своих
- Много вызовов геттеров к другому объекту
- Данные и поведение разделены

**Почему плохо:**
- Неправильное место для поведения
- Плохая инкапсуляция
- Трудно поддерживать

**Рефакторинги:**
- Перенос метода
- Перенос поля
- Извлечение метода (затем перенос)

**Пример (До):**
```javascript
class Order {
  getDiscountedPrice(customer) {
    // Активно использует данные customer
    if (customer.loyaltyYears > 5) {
      return this.price * customer.discountRate;
    }
    return this.price;
  }
}
```

**Пример (После):**
```javascript
class Customer {
  getDiscountedPriceFor(price) {
    if (this.loyaltyYears > 5) {
      return price * this.discountRate;
    }
    return price;
  }
}
```

---

### Неуместная близость

**Признаки:**
- Классы обращаются к приватным частям друг друга
- Двунаправленные ссылки
- Подклассы знают слишком много о родителях

**Почему плохо:**
- Высокая связность
- Изменения каскадируются
- Трудно изменить одно без другого

**Рефакторинги:**
- Перенос метода
- Перенос поля
- Замена двунаправленной ссылки однонаправленной
- Извлечение класса
- Скрытие делегата

---

### Цепочки сообщений

**Признаки:**
- Длинные цепочки вызовов методов: `a.getB().getC().getD().getValue()`
- Клиент зависит от навигационной структуры
- Код «крушение поезда»

**Почему плохо:**
- Хрупко — любое изменение ломает цепочку
- Нарушает закон Деметры
- Связность со структурой

**Рефакторинги:**
- Скрытие делегата
- Извлечение метода
- Перенос метода

**Пример:**
```javascript
// Плохо: Цепочка сообщений
const managerName = employee.getDepartment().getManager().getName();

// Лучше: Скрыть делегирование
const managerName = employee.getManagerName();
```

---

### Посредник

**Признаки:**
- Класс, который только делегирует другому
- Половина методов являются делегированиями
- Нет добавленной ценности

**Почему плохо:**
- Ненужная косвенность
- Накладные расходы на обслуживание
- Запутанная архитектура

**Рефакторинги:**
- Удаление посредника
- Встраивание метода

---

## Руководство по серьёзности запахов

| Серьёзность | Описание | Действие |
|------------|---------|---------|
| **Критическая** | Блокирует разработку, вызывает баги | Исправить немедленно |
| **Высокая** | Значительная нагрузка на обслуживание | Исправить в текущем спринте |
| **Средняя** | Заметно, но управляемо | Запланировать на ближайшее время |
| **Низкая** | Незначительное неудобство | Исправить оппортунистически |

## Чек-лист быстрого обнаружения

Используйте этот чек-лист при проверке кода:

- [ ] Есть ли методы > 30 строк?
- [ ] Есть ли классы > 300 строк?
- [ ] Есть ли методы с > 4 параметрами?
- [ ] Есть ли дублированные блоки кода?
- [ ] Есть ли switch/case по кодам типов?
- [ ] Есть ли неиспользуемый код?
- [ ] Есть ли методы, активно использующие данные другого класса?
- [ ] Есть ли длинные цепочки вызовов методов?
- [ ] Есть ли комментарии, объясняющие «что», а не «почему»?
- [ ] Есть ли примитивы, которые должны быть объектами?
