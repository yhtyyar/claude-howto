# Навык рефакторинга кода

Систематический подход к рефакторингу кода на основе книги Мартина Фаулера *«Рефакторинг: улучшение существующего кода»* (2-е издание). Этот навык делает акцент на безопасных, поэтапных изменениях с поддержкой тестов.

> «Рефакторинг — это процесс изменения программной системы таким образом, что не изменяется внешнее поведение кода, но улучшается его внутренняя структура.» — Мартин Фаулер

## Основные принципы

1. **Сохранение поведения**: Внешнее поведение должно оставаться неизменным
2. **Маленькие шаги**: Делать крошечные, тестируемые изменения
3. **Тест-ориентированность**: Тесты — страховочная сеть
4. **Непрерывность**: Рефакторинг — постоянный процесс, а не разовое событие
5. **Совместность**: На каждом этапе требуется одобрение пользователя

## Обзор рабочего процесса

```
Фаза 1: Исследование и анализ
    ↓
Фаза 2: Оценка покрытия тестами
    ↓
Фаза 3: Определение запахов кода
    ↓
Фаза 4: Создание плана рефакторинга
    ↓
Фаза 5: Поэтапная реализация
    ↓
Фаза 6: Ревью и итерация
```

## Фаза 1: Исследование и анализ

### Цели
- Понять структуру и цель кодовой базы
- Определить масштаб рефакторинга
- Собрать контекст о бизнес-требованиях

### Вопросы для пользователя
Перед началом уточнить:

1. **Масштаб**: Какие файлы/модули/функции нуждаются в рефакторинге?
2. **Цели**: Какие проблемы вы пытаетесь решить? (читаемость, производительность, поддерживаемость)
3. **Ограничения**: Есть ли области, которые НЕ следует изменять?
4. **Временные ограничения**: Это блокирует другую работу?
5. **Статус тестов**: Существуют ли тесты? Они проходят?

### Действия
- [ ] Прочитать и понять целевой код
- [ ] Определить зависимости и интеграции
- [ ] Задокументировать текущую архитектуру
- [ ] Отметить существующие маркеры технического долга (TODO, FIXME)

### Вывод
Представить результаты пользователю:
- Сводка структуры кода
- Выявленные проблемные области
- Первоначальные рекомендации
- **Запросить одобрение для продолжения**

## Фаза 2: Оценка покрытия тестами

### Почему тесты важны
> «Рефакторинг без тестов — как езда без ремня безопасности.» — Мартин Фаулер

Тесты — **ключевой инструмент** безопасного рефакторинга. Без них рискуете внести баги.

### Шаги оценки

1. **Проверить наличие тестов**
   ```bash
   # Найти тестовые файлы
   find . -name "*test*" -o -name "*spec*" | head -20
   ```

2. **Запустить существующие тесты**
   ```bash
   # JavaScript/TypeScript
   npm test

   # Python
   pytest -v

   # Java
   mvn test
   ```

3. **Проверить покрытие (если доступно)**
   ```bash
   # JavaScript
   npm run test:coverage

   # Python
   pytest --cov=.
   ```

### Точка принятия решения: Спросить пользователя

**Если тесты существуют и проходят:**
- Перейти к Фазе 3

**Если тесты отсутствуют или неполные:**
Предложить варианты:
1. Сначала написать тесты (рекомендуется)
2. Добавлять тесты постепенно в ходе рефакторинга
3. Продолжить без тестов (рискованно — требует подтверждения пользователя)

**Если тесты не проходят:**
- ОСТАНОВИТЬСЯ. Исправить упавшие тесты перед рефакторингом
- Спросить пользователя: Следует ли нам сначала исправить тесты?

### Руководство по написанию тестов (при необходимости)

Для каждой рефакторируемой функции убедиться, что тесты охватывают:
- Счастливый путь (нормальная работа)
- Граничные случаи (пустые входные данные, null, границы)
- Сценарии ошибок (неверные входные данные, исключения)

Использовать цикл «красный-зелёный-рефакторинг»:
1. Написать упавший тест (красный)
2. Сделать его прохождение (зелёный)
3. Провести рефакторинг

## Фаза 3: Определение запахов кода

### Что такое запахи кода?
Симптомы более глубоких проблем в коде. Это не баги, а индикаторы того, что код можно улучшить.

### Распространённые запахи кода для проверки

Полный каталог см. в [references/code-smells.md](references/code-smells.md).

#### Краткий справочник

| Запах | Признаки | Влияние |
|-------|---------|---------|
| **Длинный метод** | Методы > 30-50 строк | Трудно понять, протестировать, поддерживать |
| **Дублированный код** | Одна логика в нескольких местах | Исправления баги нужно делать в нескольких местах |
| **Большой класс** | Класс со слишком многими обязанностями | Нарушает принцип единственной ответственности |
| **Зависть к функциям** | Метод использует данные другого класса больше | Плохая инкапсуляция |
| **Примитивная одержимость** | Злоупотребление примитивами вместо объектов | Отсутствующие доменные концепции |
| **Длинный список параметров** | Методы с 4+ параметрами | Трудно вызвать правильно |
| **Группы данных** | Одни и те же данные, появляющиеся вместе | Отсутствующая абстракция |
| **Switch-операторы** | Сложные switch/if-else цепочки | Трудно расширять |
| **Спекулятивная обобщённость** | Код «на всякий случай» | Ненужная сложность |
| **Мёртвый код** | Неиспользуемый код | Путаница, бремя обслуживания |

### Шаги анализа

1. **Автоматизированный анализ** (при наличии скриптов)
   ```bash
   python scripts/detect-smells.py <файл>
   ```

2. **Ручной просмотр**
   - Систематически просмотреть код
   - Отметить каждый запах с местоположением и серьёзностью
   - Категоризировать по влиянию (Критический/Высокий/Средний/Низкий)

3. **Приоритизация**
   Сфокусироваться на запахах, которые:
   - Блокируют текущую разработку
   - Вызывают баги или путаницу
   - Затрагивают наиболее часто изменяемые пути кода

### Вывод: Отчёт о запахах

Представить пользователю:
- Список выявленных запахов с местоположениями
- Оценка серьёзности каждого
- Рекомендуемый порядок приоритетов
- **Запросить одобрение приоритетов**

## Фаза 4: Создание плана рефакторинга

### Выбор рефакторингов

Для каждого запаха выбрать подходящий рефакторинг из каталога.

Полный список см. в [references/refactoring-catalog.md](references/refactoring-catalog.md).

#### Соответствие запах → рефакторинг

| Запах кода | Рекомендуемые рефакторинги |
|------------|--------------------------|
| Длинный метод | Извлечение метода, Замена временной переменной запросом |
| Дублированный код | Извлечение метода, Подъём метода, Шаблонный метод |
| Большой класс | Извлечение класса, Извлечение подкласса |
| Зависть к функциям | Перенос метода, Перенос поля |
| Примитивная одержимость | Замена примитива объектом, Замена кода типа классом |
| Длинный список параметров | Введение объекта-параметра, Сохранение целого объекта |
| Группы данных | Извлечение класса, Введение объекта-параметра |
| Switch-операторы | Замена условного полиморфизмом |
| Спекулятивная обобщённость | Свёртывание иерархии, Встраивание класса, Удаление мёртвого кода |
| Мёртвый код | Удаление мёртвого кода |

### Структура плана

Использовать шаблон в [templates/refactoring-plan.md](templates/refactoring-plan.md).

Для каждого рефакторинга:
1. **Цель**: Какой код изменится
2. **Запах**: Какую проблему устраняет
3. **Рефакторинг**: Какую технику применить
4. **Шаги**: Детализированные микро-шаги
5. **Риски**: Что может пойти не так
6. **Откат**: Как отменить при необходимости

### Поэтапный подход

**КРИТИЧНО**: Вводить рефакторинг постепенно по фазам.

**Фаза A: Быстрые победы** (Низкий риск, высокая ценность)
- Переименование переменных для ясности
- Извлечение очевидного дублированного кода
- Удаление мёртвого кода

**Фаза B: Структурные улучшения** (Средний риск)
- Извлечение методов из длинных функций
- Введение объектов-параметров
- Перенос методов в подходящие классы

**Фаза C: Архитектурные изменения** (Повышенный риск)
- Замена условных операторов полиморфизмом
- Извлечение классов
- Введение паттернов проектирования

### Точка принятия решения: Представить план пользователю

Перед реализацией:
- Показать полный план рефакторинга
- Объяснить каждую фазу и её риски
- Получить явное одобрение для каждой фазы
- **Спросить**: «Приступить к Фазе A?»

## Фаза 5: Поэтапная реализация

### Золотое правило
> «Изменение → Тест → Зелёный? → Коммит → Следующий шаг»

### Ритм реализации

Для каждого шага рефакторинга:

1. **Предварительная проверка**
   - Тесты проходят (зелёный)
   - Код компилируется

2. **Сделать ОДНО маленькое изменение**
   - Следовать механике из каталога
   - Минимизировать изменения

3. **Верификация**
   - Немедленно запустить тесты
   - Проверить ошибки компиляции

4. **Если тесты проходят (зелёный)**
   - Зафиксировать с описательным сообщением
   - Перейти к следующему шагу

5. **Если тесты не проходят (красный)**
   - НЕМЕДЛЕННО ОСТАНОВИТЬСЯ
   - Отменить изменение
   - Проанализировать, что пошло не так
   - Спросить пользователя если непонятно

### Стратегия коммитов

Каждый коммит должен быть:
- **Атомарным**: Одно логическое изменение
- **Обратимым**: Легко откатить
- **Описательным**: Понятное сообщение коммита

Примеры сообщений коммитов:
```
refactor: Извлечь calculateTotal() из processOrder()
refactor: Переименовать 'x' в 'customerCount' для ясности
refactor: Удалить неиспользуемый метод validateOldFormat()
```

### Отчётность о прогрессе

После каждой подфазы, доложить пользователю:
- Сделанные изменения
- Тесты всё ещё проходят?
- Возникшие проблемы
- **Спросить**: «Продолжить со следующей партией?»

## Фаза 6: Ревью и итерация

### Чек-лист после рефакторинга

- [ ] Все тесты проходят
- [ ] Нет новых предупреждений/ошибок
- [ ] Код успешно компилируется
- [ ] Поведение не изменилось (ручная верификация)
- [ ] Документация обновлена при необходимости
- [ ] История коммитов чистая

### Сравнение метрик

Запустить анализ сложности до и после:
```bash
python scripts/analyze-complexity.py <файл>
```

Представить улучшения:
- Изменение количества строк кода
- Изменение цикломатической сложности
- Изменение индекса поддерживаемости

### Ревью пользователя

Представить финальные результаты:
- Сводка всех изменений
- Сравнение кода до/после
- Улучшения метрик
- Оставшийся технический долг
- **Спросить**: «Довольны ли вы этими изменениями?»

### Следующие шаги

Обсудить с пользователем:
- Дополнительные запахи для устранения?
- Запланировать следующий рефакторинг?
- Применить похожие изменения в другом месте?

## Важные руководящие принципы

### Когда ОСТАНОВИТЬСЯ и спросить

Всегда останавливаться и консультироваться с пользователем когда:
- Неуверены в бизнес-логике
- Изменение может затронуть внешние API
- Покрытие тестами недостаточно
- Нужно принять значительное архитектурное решение
- Уровень риска повышается
- Вы столкнулись с неожиданной сложностью

### Правила безопасности

1. **Никогда не рефакторить без тестов** (если только пользователь явно не признаёт риск)
2. **Никогда не делать больших изменений** — разбивать на крошечные шаги
3. **Никогда не пропускать запуск тестов** после каждого изменения
4. **Никогда не продолжать при упавших тестах** — сначала исправить или откатить
5. **Никогда не предполагать** — при сомнении, спрашивать

### Что НЕ делать

- Не совмещать рефакторинг с добавлением функций
- Не рефакторить во время production-аварий
- Не рефакторить код, который вы не понимаете
- Не переусложнять — держать в простоте
- Не рефакторить всё сразу

## Пример быстрого старта

### Сценарий: Длинный метод с дублированием

**До:**
```javascript
function processOrder(order) {
  // 150 строк кода с:
  // - Дублированной логикой валидации
  // - Встроенными вычислениями
  // - Смешанными обязанностями
}
```

**Шаги рефакторинга:**

1. **Убедиться, что тесты существуют** для processOrder()
2. **Извлечь** валидацию в validateOrder()
3. **Тест** — должен пройти
4. **Извлечь** вычисление в calculateOrderTotal()
5. **Тест** — должен пройти
6. **Извлечь** уведомление в notifyCustomer()
7. **Тест** — должен пройти
8. **Ревью** — processOrder() теперь оркестрирует 3 чёткие функции

**После:**
```javascript
function processOrder(order) {
  validateOrder(order);
  const total = calculateOrderTotal(order);
  notifyCustomer(order, total);
  return { order, total };
}
```
