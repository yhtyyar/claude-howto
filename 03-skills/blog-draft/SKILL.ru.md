description: Написать черновик блог-поста из идей и ресурсов. Использовать когда пользователи хотят написать блог-пост, создать контент из исследования или написать статью. Проводит через исследование, мозговой штурм, создание структуры и итеративный черновик с контролем версий.
---

## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ОБЯЗАНЫ** учитывать ввод пользователя перед началом работы. Пользователь должен предоставить:
- **Идея/Тема**: Основная концепция или тема для блог-поста
- **Ресурсы**: URL, файлы или ссылки для исследования (опционально, но рекомендуется)
- **Целевая аудитория**: Для кого блог-пост (опционально)
- **Тон/Стиль**: Формальный, неформальный, технический и т.д. (опционально)

**ВАЖНО**: Если пользователь запрашивает обновления **существующего блог-поста**, пропустить шаги 0-8 и начать сразу с **Шага 9**. Сначала прочитать существующие файлы черновика, затем продолжить процесс итерации.

## Поток выполнения

Следовать этим шагам последовательно. **Не пропускать шаги и не продолжать без одобрения пользователя там, где это указано.**

### Шаг 0: Создать папку проекта

1. Сгенерировать имя папки в формате: `ГГГГ-ММ-ДД-краткое-название-темы`
   - Использовать сегодняшнюю дату
   - Создать короткий, дружественный для URL слаг из темы (строчные буквы, дефисы, макс. 5 слов)

2. Создать структуру папки:
   ```
   blog-posts/
   └── ГГГГ-ММ-ДД-краткое-название-темы/
       └── resources/
   ```

3. Подтвердить создание папки с пользователем перед продолжением.

### Шаг 1: Исследование и сбор ресурсов

1. Создать подпапку `resources/` в директории блог-поста

2. Для каждого предоставленного ресурса:
   - **URL**: Получить и сохранить ключевую информацию в `resources/` как markdown-файлы
   - **Файлы**: Прочитать и обобщить в `resources/`
   - **Темы**: Использовать веб-поиск для сбора актуальной информации

3. Для каждого ресурса создать файл сводки в `resources/`:
   - `resources/source-1-[краткое-имя].md`
   - `resources/source-2-[краткое-имя].md`
   - и т.д.

4. Каждая сводка должна включать:
   ```markdown
   # Источник: [Заголовок/URL]

   ## Ключевые моменты
   - Момент 1
   - Момент 2

   ## Релевантные цитаты/данные
   - Цитата или статистика 1
   - Цитата или статистика 2

   ## Как это относится к теме
   Краткое объяснение актуальности
   ```

5. Представить сводку исследования пользователю.

### Шаг 2: Мозговой штурм и уточнение

1. На основе идеи и исследованных ресурсов представить:
   - **Основные темы**, выявленные из исследования
   - **Потенциальные углы** для блог-поста
   - **Ключевые моменты**, которые должны быть освещены
   - **Пробелы** в информации, требующие уточнения

2. Задать уточняющие вопросы:
   - Какой главный вывод должны сделать читатели?
   - Есть ли конкретные моменты из исследования, которые нужно выделить?
   - Какова целевая длина? (короткая: 500-800 слов, средняя: 1000-1500, длинная: 2000+)
   - Есть ли моменты, которые нужно исключить?

3. **Ждать ответов пользователя перед продолжением.**

### Шаг 3: Предложить структуру

1. Создать структурированный план:

   ```markdown
   # Структура блог-поста: [Заголовок]

   ## Мета-информация
   - **Целевая аудитория**: [кто]
   - **Тон**: [стиль]
   - **Целевая длина**: [количество слов]
   - **Главный вывод**: [ключевое сообщение]

   ## Предлагаемая структура

   ### Зацепка/Введение
   - Идея открывающей зацепки
   - Задание контекста
   - Тезис

   ### Раздел 1: [Заголовок]
   - Ключевой момент A
   - Ключевой момент B
   - Подтверждающие данные из [источника]

   ### Раздел 2: [Заголовок]
   - Ключевой момент A
   - Ключевой момент B

   [Продолжить для всех разделов...]

   ### Заключение
   - Сводка ключевых моментов
   - Призыв к действию или финальная мысль

   ## Источники для цитирования
   - Источник 1
   - Источник 2
   ```

2. Представить план пользователю и **попросить одобрения или изменений**.

### Шаг 4: Сохранить одобренную структуру

1. После одобрения пользователем сохранить план в `OUTLINE.md` в папке блог-поста.

2. Подтвердить сохранение плана.

### Шаг 5: Зафиксировать план (если в git-репозитории)

1. Проверить, является ли текущая директория git-репозиторием.

2. Если да:
   - Зафиксировать новые файлы: папку блог-поста, ресурсы и OUTLINE.md
   - Создать коммит с сообщением: `docs: Add outline for blog post - [имя-темы]`
   - Запушить на удалённый сервер

3. Если не git-репо, пропустить этот шаг и сообщить пользователю.

### Шаг 6: Написать черновик

1. На основе одобренной структуры написать полный черновик блог-поста.

2. Строго следовать структуре из OUTLINE.md.

3. Включить:
   - Привлекательное введение с зацепкой
   - Чёткие заголовки разделов
   - Подтверждающие данные и примеры из исследования
   - Плавные переходы между разделами
   - Сильное заключение с выводом
   - **Цитирование**: Все сравнения, статистика, данные и фактические утверждения ДОЛЖНЫ цитировать оригинальный источник

4. Сохранить черновик как `draft-v0.1.md` в папке блог-поста.

5. Формат:
   ```markdown
   # [Заголовок блог-поста]

   *[Опционально: подзаголовок или слоган]*

   [Полный контент со встроенными цитатами...]

   ---

   ## Источники
   - [1] Название источника 1 - URL или цитата
   - [2] Название источника 2 - URL или цитата
   - [3] Название источника 3 - URL или цитата
   ```

6. **Требования к цитированию**:
   - Каждая точка данных, статистика или сравнение ДОЛЖНЫ иметь встроенную цитату
   - Использовать нумерованные ссылки [1], [2] и т.д., или именованные цитаты [Название источника]
   - Связывать цитаты с разделом Источники в конце
   - Пример: «Исследования показывают, что 65% разработчиков предпочитают TypeScript [1]»

### Шаг 7: Зафиксировать черновик (если в git-репозитории)

1. Проверить наличие git-репозитория.

2. Если да:
   - Зафиксировать файл черновика
   - Создать коммит с сообщением: `docs: Add draft v0.1 for blog post - [имя-темы]`
   - Запушить на удалённый сервер

3. Если нет — пропустить и сообщить пользователю.

### Шаг 8: Представить черновик для ревью

1. Представить контент черновика пользователю.

2. Попросить обратную связь:
   - Общее впечатление?
   - Разделы, требующие расширения или сокращения?
   - Необходимые корректировки тона?
   - Недостающая информация?
   - Конкретные правки или переработки?

3. **Ждать ответа пользователя.**

### Шаг 9: Итерация или финализация

**Если пользователь запрашивает изменения:**
1. Записать все запрошенные изменения
2. Вернуться к Шагу 6 со следующими корректировками:
   - Увеличить номер версии (v0.2, v0.3 и т.д.)
   - Включить всю обратную связь
   - Сохранить как `draft-v[X.Y].md`
   - Повторить Шаги 7-8

**Если пользователь одобряет:**
1. Подтвердить финальную версию черновика
2. Опционально переименовать в `final.md` по запросу пользователя
3. Обобщить процесс создания блог-поста:
   - Всего созданных версий
   - Ключевые изменения между версиями
   - Финальное количество слов
   - Созданные файлы

## Отслеживание версий

Все черновики сохраняются с инкрементальной нумерацией версий:
- `draft-v0.1.md` — Первоначальный черновик
- `draft-v0.2.md` — После первого раунда обратной связи
- `draft-v0.3.md` — После второго раунда обратной связи
- и т.д.

Это позволяет отслеживать эволюцию блог-поста и откатываться при необходимости.

## Структура выходных файлов

```
blog-posts/
└── ГГГГ-ММ-ДД-имя-темы/
    ├── resources/
    │   ├── source-1-name.md
    │   ├── source-2-name.md
    │   └── ...
    ├── OUTLINE.md
    ├── draft-v0.1.md
    ├── draft-v0.2.md (при итерациях)
    └── draft-v0.3.md (при дополнительных итерациях)
```

## Советы по качеству

- **Зацепка**: Начинать с вопроса, удивительного факта или понятного сценария
- **Поток**: Каждый абзац должен быть связан со следующим
- **Доказательства**: Подтверждать утверждения данными из исследования
- **Цитирование**: ВСЕГДА цитировать источники для всей статистики, сравнений и фактических утверждений
- **Голос**: Поддерживать согласованный тон на протяжении всего текста
- **Длина**: Соблюдать целевое количество слов
- **Читаемость**: Использовать короткие абзацы, маркированные списки где уместно
- **Призыв к действию**: Завершить чётким призывом к действию или провокационным вопросом

## Примечания

- Всегда ждать одобрения пользователя в указанных контрольных точках
- Сохранять все версии черновиков для истории
- Использовать веб-поиск для актуальной информации при предоставлении URL
- Если ресурсов недостаточно, попросить пользователя о большем или предложить дополнительные исследования
- Адаптировать тон к целевой аудитории (техническая, широкая, деловая и т.д.)
