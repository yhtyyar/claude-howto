# Политика безопасности

## Обзор

Безопасность проекта Claude How To важна для нас. Этот документ описывает наши практики безопасности и объясняет, как ответственно сообщать об уязвимостях безопасности.

## Поддерживаемые версии

Мы предоставляем обновления безопасности для следующих версий:

| Версия | Статус | Поддержка до |
|---------|--------|---------------|
| Последняя (main) | ✅ Активная | Текущая + 6 месяцев |
| Релизы 1.x | ✅ Активная | До следующей мажорной версии |

**Примечание**: Как образовательный проект-руководство, мы фокусируемся на поддержании актуальных лучших практик и безопасности документации, а не на традиционной поддержке версий. Обновления применяются напрямую к ветке main.

## Практики безопасности

### Безопасность кода

1. **Управление зависимостями**
   - Все Python-зависимости зафиксированы в `requirements.txt`
   - Регулярные обновления через dependabot и ручной обзор
   - Сканирование безопасности с Bandit на каждый коммит
   - Pre-commit хуки для проверок безопасности

2. **Качество кода**
   - Линтинг с Ruff выявляет типичные проблемы
   - Проверка типов с mypy предотвращает уязвимости, связанные с типами
   - Pre-commit хуки обеспечивают стандарты
   - Все изменения ревьюируются перед слиянием

3. **Контроль доступа**
   - Защита ветки на ветке `main`
   - Требуемые ревью перед слиянием
   - Статус-проверки должны проходить перед слиянием
   - Ограниченный доступ на запись к репозиторию

### Безопасность документации

1. **Никаких секретов в примерах**
   - Все API-ключи в примерах — плейсхолдеры
   - Учётные данные никогда не хардкодятся
   - Файлы `.env.example` показывают требуемые переменные
   - Чёткие предупреждения об управлении секретами

2. **Лучшие практики безопасности**
   - Примеры демонстрируют безопасные паттерны
   - Предупреждения безопасности выделены в документации
   - Ссылки на официальные руководства по безопасности
   - Обработка учётных данных обсуждается в соответствующих разделах

3. **Обзор контента**
   - Вся документация проверяется на вопросы безопасности
   - Аспекты безопасности в руководстве по внесению вклада
   - Валидация внешних ссылок и ссылок

### Безопасность зависимостей

1. **Сканирование**
   - Bandit сканирует весь Python-код на уязвимости
   - Проверка уязвимостей зависимостей через GitHub security alerts
   - Регулярные ручные аудиты безопасности

2. **Обновления**
   - Патчи безопасности применяются оперативно
   - Мажорные версии оцениваются тщательно
   - Changelog включает обновления, связанные с безопасностью

3. **Прозрачность**
   - Обновления безопасности документируются в коммитах
   - Раскрытие уязвимостей обрабатывается ответственно
   - Публичные рекомендации по безопасности, когда уместно

## Сообщение об уязвимости

### Вопросы безопасности, которые нас интересуют

Мы ценим отчёты о:
- **Уязвимостях кода** в скриптах или примерах
- **Уязвимостях зависимостей** в Python-пакетах
- **Криптографических проблемах** в любых примерах кода
- **Недостатках аутентификации/авторизации** в документации
- **Рисках утечки данных** в примерах конфигурации
- **Инъекционных уязвимостях** (SQL, command и т.д.)
- **Проблемах SSRF/XXE/Path traversal**

### Вопросы безопасности вне области

Это вне области применения данного проекта:
- Уязвимости в самом Claude Code (сообщайте Anthropic)
- Проблемы с внешними сервисами или библиотеками (сообщайте upstream)
- Социальная инженерия или образование пользователей (не применимо к этому руководству)
- Теоретические уязвимости без proof of concept
- Уязвимости в зависимостях, о которых сообщают через официальные каналы

## Как сообщить

### Приватный репортинг (предпочтительно)

**Для чувствительных вопросов безопасности, пожалуйста, используйте приватный репортинг уязвимостей GitHub:**

1. Посетите: https://github.com/luongnv89/claude-howto/security/advisories
2. Нажмите "Report a vulnerability"
3. Заполните детали уязвимости
4. Включите:
   - Чёткое описание уязвимости
   - Затронутый компонент (файл, раздел, пример)
   - Потенциальное влияние
   - Шаги для воспроизведения (если применимо)
   - Предлагаемое исправление (если есть)

**Что происходит дальше:**
- Мы подтвердим получение в течение 48 часов
- Мы исследуем и оценим серьёзность
- Мы будем работать с вами над разработкой исправления
- Мы согласуем время раскрытия
- Мы отдадим вам заслуги в рекомендации по безопасности (если вы не предпочитаете анонимность)

### Публичный репортинг

Для нечувствительных вопросов или уже публичных:

1. **Откройте GitHub Issue** с лейблом `security`
2. Включите:
   - Заголовок: `[SECURITY]` с последующим кратким описанием
   - Детальное описание
   - Затронутый файл или раздел
   - Потенциальное влияние
   - Предлагаемое исправление

## Процесс реагирования на уязвимости

### Оценка (24 часа)

1. Мы подтверждаем получение отчёта
2. Мы оцениваем серьёзность с использованием [CVSS v3.1](https://www.first.org/cvss/v3.1/specification-document)
3. Мы определяем, входит ли в область
4. Мы связываемся с вами с начальной оценкой

### Разработка (1-7 дней)

1. Мы разрабатываем исправление
2. Мы проверяем и тестируем исправление
3. Мы создаём рекомендацию по безопасности
4. Мы готовим примечания к релизу

### Раскрытие (варьируется по серьёзности)

**Критическая (CVSS 9.0-10.0)**
- Исправление выпускается немедленно
- Выпускается публичная рекомендация
- 24-часовое предварительное уведомление репортёрам

**Высокая (CVSS 7.0-8.9)**
- Исправление выпускается в течение 48-72 часов
- 5-дневное предварительное уведомление репортёрам
- Публичная рекомендация при релизе

**Средняя (CVSS 4.0-6.9)**
- Исправление выпускается в следующем регулярном обновлении
- Публичная рекомендация при релизе

**Низкая (CVSS 0.1-3.9)**
- Исправление включено в следующее регулярное обновление
- Рекомендация при релизе

### Публикация

Мы публикуем рекомендации по безопасности, включающие:
- Описание уязвимости
- Затронутые компоненты
- Оценка серьёзности (CVSS score)
- Исправленную версию
- Обходные пути (если применимо)
- Заслуги репортёру (с разрешения)

## Лучшие практики для репортёров

### Перед репортингом

- **Проверьте вопрос**: Можете ли вы воспроизвести его консистентно?
- **Поищите существующие задачи**: Уже ли оно сообщалось?
- **Проверьте документацию**: Есть ли руководство по безопасному использованию?
- **Протестируйте исправление**: Работает ли ваше предлагаемое исправление?

### При репортинге

- **Будьте конкретны**: Предоставьте точные пути файлов и номера строк
- **Включите контекст**: Почему это вопрос безопасности?
- **Покажите влияние**: Что может сделать атакующий?
- **Предоставьте шаги**: Как мы можем воспроизвести?
- **Предлагайте исправления**: Как бы вы исправили?

### После репортинга

- **Будьте терпеливы**: У нас ограниченные ресурсы
- **Будьте отзывчивы**: Отвечайте на последующие вопросы быстро
- **Держите конфиденциально**: Не раскрывайте публично до исправления
- **Уважайте координацию**: Следуйте нашему таймлайну для раскрытия

## Заголовки безопасности и конфигурация

### Безопасность репозитория

- **Защита ветки**: Для изменений в ветке main требуется 2 ревью
- **Статус-проверки**: Все CI/CD проверки должны проходить
- **CODEOWNERS**: Назначенные ревьюверы для ключевых файлов
- **Подписанные коммиты**: Рекомендуется для контрибьюторов

### Безопасность разработки

```bash
# Установить pre-commit хуки
pre-commit install

# Запустить сканирование безопасности локально

mypy scripts/ --ignore-missing-imports
ruff check scripts/
```

### Безопасность зависимостей

```bash
# Проверить известные уязвимости
pip install safety

# Или используйте pip-audit
pip install pip-audit
pip-audit
```

## Руководства по безопасности для контрибьюторов

### При написании примеров

1. **Никогда не хардкодьте секреты**
   ```python
   # ❌ Плохо
   api_key = "sk-1234567890"

   # ✅ Хорошо
   api_key = os.getenv("API_KEY")
   ```

2. **Предупреждайте о последствиях для безопасности**
   ```markdown
   ⚠️ **Примечание по безопасности**: Никогда не коммитьте файлы `.env` в git.
   Добавьте в `.gitignore` немедленно.
   ```

3. **Используйте безопасные настройки по умолчанию**
   - Включайте аутентификацию по умолчанию
   - Используйте HTTPS, где применимо
   - Валидируйте и санитизируйте вводы
   - Используйте параметризованные запросы

4. **Документируйте аспекты безопасности**
   - Объясняйте, почему безопасность важна
   - Показывайте безопасные и небезопасные паттерны
   - Ссылайтесь на авторитетные источники
   - Включайте предупреждения заметно

### При ревью вкладов

1. **Проверяйте на обнаруженные секреты**
   - Сканируйте на типичные паттерны (api_key=, password=)
   - Ревьюйте конфигурационные файлы
   - Проверяйте переменные окружения

2. **Проверяйте практики безопасного кодирования**
   - Нет хардкодированных учётных данных
   - Правильная валидация ввода
   - Безопасная аутентификация/авторизация
   - Безопасная обработка файлов

3. **Тестируйте последствия для безопасности**
   - Может ли это быть использовано неправильно?
   - Что худший случай?
   - Есть ли крайние случаи?
