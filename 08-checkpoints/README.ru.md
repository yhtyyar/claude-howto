# Контрольные точки и перемотка

Контрольные точки позволяют сохранять состояние разговора и возвращаться к предыдущим моментам в сессии Claude Code. Это незаменимо для исследования различных подходов, восстановления после ошибок или сравнения альтернативных решений.

## Обзор

Контрольные точки позволяют сохранять состояние разговора и возвращаться к предыдущим моментам, обеспечивая безопасные эксперименты и исследование множества подходов. Это снимки состояния разговора, включающие:
- Все обменянные сообщения
- Сделанные изменения файлов
- Историю использования инструментов
- Контекст сессии

Контрольные точки незаменимы при исследовании различных подходов, восстановлении после ошибок или сравнении альтернативных решений.

## Ключевые концепции

| Концепция | Описание |
|-----------|----------|
| **Контрольная точка** | Снимок состояния разговора, включая сообщения, файлы и контекст |
| **Перемотка** | Возврат к предыдущей контрольной точке с отменой последующих изменений |
| **Точка ветвления** | Контрольная точка, из которой исследуются несколько подходов |

## Доступ к контрольным точкам

Получить доступ к контрольным точкам можно двумя основными способами:

### Горячая клавиша
Нажми `Esc` дважды (`Esc` + `Esc`), чтобы открыть интерфейс контрольных точек и просмотреть сохранённые контрольные точки.

### Слэш-команда
Используй команду `/rewind` (псевдоним: `/checkpoint`) для быстрого доступа:

```bash
# Открыть интерфейс перемотки
/rewind

# Или использовать псевдоним
/checkpoint
```

## Варианты перемотки

При перемотке появляется меню из пяти вариантов:

1. **Восстановить код и разговор** — Вернуть как файлы, так и сообщения к этой контрольной точке
2. **Восстановить разговор** — Перемотать только сообщения, оставить текущий код как есть
3. **Восстановить код** — Вернуть только изменения файлов, оставить полную историю разговора
4. **Резюмировать отсюда** — Сжать разговор с этого момента в AI-сгенерированное резюме вместо его удаления. Исходные сообщения сохраняются в транскрипте. При желании можно указать инструкции для фокусировки резюме на конкретных темах.
5. **Отмена** — Отменить и вернуться к текущему состоянию

## Автоматические контрольные точки

Claude Code автоматически создаёт контрольные точки за тебя:

- **Каждый пользовательский промпт** — Новая контрольная точка создаётся с каждым вводом пользователя
- **Постоянство** — Контрольные точки сохраняются между сессиями
- **Авто-очистка** — Контрольные точки автоматически очищаются через 30 дней

Это означает, что всегда можно вернуться к любому предыдущему моменту разговора — от нескольких минут назад до нескольких дней назад.

## Сценарии использования

| Сценарий | Рабочий процесс |
|----------|-----------------|
| **Исследование подходов** | Сохранить → Попробовать А → Сохранить → Перемотать → Попробовать Б → Сравнить |
| **Безопасный рефакторинг** | Сохранить → Рефакторинг → Тест → Если не прошёл: Перемотать |
| **A/B-тестирование** | Сохранить → Дизайн А → Сохранить → Перемотать → Дизайн Б → Сравнить |
| **Восстановление после ошибки** | Обнаружить проблему → Перемотать к последнему хорошему состоянию |

## Использование контрольных точек

### Просмотр и перемотка

Нажми `Esc` дважды или используй `/rewind` для открытия браузера контрольных точек. Увидишь список всех доступных контрольных точек с временными метками. Выбери любую контрольную точку для перемотки к этому состоянию.

### Детали контрольной точки

Каждая контрольная точка показывает:
- Временную метку создания
- Изменённые файлы
- Количество сообщений в разговоре
- Использованные инструменты

## Практические примеры

### Пример 1: Исследование различных подходов

```
Пользователь: Добавим слой кэширования к API

Claude: Добавлю Redis-кэширование в конечные точки API...
[Вносит изменения в контрольной точке А]

Пользователь: Давайте попробуем in-memory кэширование вместо этого

Claude: Перемотаю для исследования другого подхода...
[Пользователь нажимает Esc+Esc и перематывает к контрольной точке А]
[Реализует in-memory кэширование в контрольной точке Б]

Пользователь: Теперь могу сравнить оба подхода
```

### Пример 2: Восстановление после ошибок

```
Пользователь: Отрефакторь модуль аутентификации для использования JWT

Claude: Отрефакторю модуль аутентификации...
[Вносит обширные изменения]

Пользователь: Подождите, это сломало интеграцию OAuth. Давайте вернёмся назад.

Claude: Помогу перемотать к состоянию до рефакторинга...
[Пользователь нажимает Esc+Esc и выбирает контрольную точку до рефакторинга]

Пользователь: Давайте попробуем более консервативный подход
```

### Пример 3: Безопасные эксперименты

```
Пользователь: Попробуем переписать это в функциональном стиле
[Создаётся контрольная точка перед экспериментом]

Claude: [Вносит экспериментальные изменения]

Пользователь: Тесты не проходят. Перемотаем.
[Пользователь нажимает Esc+Esc и перематывает к контрольной точке]

Claude: Я перемотал изменения. Попробуем другой подход.
```

### Пример 4: Ветвление подходов

```
Пользователь: Хочу сравнить два дизайна базы данных
[Берём на заметку контрольную точку — назовём её «Начало»]

Claude: Создам первый дизайн...
[Реализует схему А]

Пользователь: Теперь вернёмся и попробуем второй подход
[Пользователь нажимает Esc+Esc и перематывает к «Началу»]

Claude: Теперь реализую схему Б...
[Реализует схему Б]

Пользователь: Отлично! Теперь у меня есть обе схемы для выбора
```

## Срок хранения контрольных точек

Claude Code автоматически управляет контрольными точками:

- Контрольные точки создаются автоматически с каждым пользовательским промптом
- Старые контрольные точки хранятся до 30 дней
- Контрольные точки очищаются автоматически, чтобы предотвратить неограниченный рост хранилища

## Паттерны рабочих процессов

### Стратегия ветвления для исследования

При исследовании нескольких подходов:

```
1. Начальная реализация → Контрольная точка А
2. Попробовать Подход 1 → Контрольная точка Б
3. Перемотать к Контрольной точке А
4. Попробовать Подход 2 → Контрольная точка В
5. Сравнить результаты из Б и В
6. Выбрать лучший подход и продолжить
```

### Паттерн безопасного рефакторинга

При внесении значительных изменений:

```
1. Текущее состояние → Контрольная точка (авто)
2. Начать рефакторинг
3. Запустить тесты
4. Если тесты прошли → Продолжить работу
5. Если тесты не прошли → Перемотать и попробовать другой подход
```

## Лучшие практики

Поскольку контрольные точки создаются автоматически, можно сосредоточиться на работе без беспокойства о ручном сохранении состояния. Однако имей в виду следующие практики:

### Эффективное использование контрольных точек

✅ **Делай:**
- Проверяй доступные контрольные точки перед перемоткой
- Используй перемотку, когда хочешь исследовать разные направления
- Сохраняй контрольные точки для сравнения разных подходов
- Понимай, что делает каждый вариант перемотки (восстановить код и разговор, восстановить разговор, восстановить код, или резюмировать)

❌ **Не делай:**
- Не полагайся только на контрольные точки для сохранения кода
- Не ожидай, что контрольные точки отслеживают внешние изменения файловой системы
- Не используй контрольные точки как замену git-коммитов

## Конфигурация

Можно переключить автоматические контрольные точки в настройках:

```json
{
  "autoCheckpoint": true
}
```

- `autoCheckpoint`: Включить или отключить автоматическое создание контрольных точек с каждым пользовательским промптом (по умолчанию: `true`)

## Ограничения

Контрольные точки имеют следующие ограничения:

- **Изменения bash-команд НЕ отслеживаются** — Операции вроде `rm`, `mv`, `cp` в файловой системе не захватываются в контрольных точках
- **Внешние изменения НЕ отслеживаются** — Изменения, сделанные вне Claude Code (в редакторе, терминале и т.д.), не захватываются
- **Не замена контролю версий** — Используй git для постоянных, проверяемых изменений в кодовой базе

## Устранение неполадок

### Отсутствующие контрольные точки

**Проблема**: Ожидаемая контрольная точка не найдена

**Решение**:
- Проверь, не были ли контрольные точки очищены
- Убедись, что `autoCheckpoint` включён в настройках
- Проверь дисковое пространство

### Перемотка не удалась

**Проблема**: Невозможно перемотать к контрольной точке

**Решение**:
- Убедись, что незафиксированные изменения не создают конфликтов
- Проверь, не повреждена ли контрольная точка
- Попробуй перемотать к другой контрольной точке

## Интеграция с Git

Контрольные точки дополняют (но не заменяют) git:

| Функция | Git | Контрольные точки |
|---------|-----|-------------------|
| Область | Файловая система | Разговор + файлы |
| Постоянство | Постоянное | Основано на сессии |
| Гранулярность | Коммиты | Любой момент |
| Скорость | Медленнее | Мгновенно |
| Совместное использование | Да | Ограниченно |

Используй оба вместе:
1. Используй контрольные точки для быстрых экспериментов
2. Используй git-коммиты для финальных изменений
3. Создавай контрольную точку перед git-операциями
4. Фиксируй успешные состояния контрольных точек в git

## Руководство по быстрому старту

### Базовый рабочий процесс

1. **Работай как обычно** — Claude Code создаёт контрольные точки автоматически
2. **Хочешь вернуться назад?** — Нажми `Esc` дважды или используй `/rewind`
3. **Выбери контрольную точку** — Выбери из списка для перемотки
4. **Выбери, что восстанавливать** — Выбери из: восстановить код и разговор, восстановить разговор, восстановить код, резюмировать отсюда, или отменить
5. **Продолжай работу** — Ты вернулся к этому моменту

### Горячие клавиши

- **`Esc` + `Esc`** — Открыть браузер контрольных точек
- **`/rewind`** — Альтернативный способ доступа к контрольным точкам
- **`/checkpoint`** — Псевдоним для `/rewind`

## Знай, когда перематывать: Мониторинг контекста

Контрольные точки позволяют вернуться назад — но как понять *когда* это нужно? По мере роста разговора контекстное окно Claude заполняется и качество модели незаметно деградирует. Можно отправлять код от «полуслепой» модели, не осознавая этого.

**[cc-context-stats](https://github.com/luongnv89/cc-context-stats)** решает эту проблему, добавляя **зоны контекста** в реальном времени в строку статуса Claude Code. Отслеживает, где ты находишься в контекстном окне — от **Plan** (зелёный, безопасно для планирования и кодирования) через **Code** (жёлтый, избегай начинать новые планы) до **Dump** (оранжевый, завершай и перематывай). Когда видишь смену зоны — время создать контрольную точку и начать заново вместо того, чтобы продолжать с деградировавшим выводом.

## Резюме

Контрольные точки — автоматическая функция Claude Code, позволяющая безопасно исследовать различные подходы без страха потерять работу. Каждый пользовательский промпт автоматически создаёт новую контрольную точку, поэтому всегда можно вернуться к любому предыдущему моменту в сессии.

Ключевые преимущества:
- Экспериментируй смело с несколькими подходами
- Быстро восстанавливайся после ошибок
- Сравнивай разные решения бок о бок
- Безопасно интегрируйся с системами контроля версий

Помни: контрольные точки не заменяют git. Используй контрольные точки для быстрых экспериментов, а git — для постоянных изменений кода.
