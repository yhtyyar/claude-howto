# Коммит и пуш всего

⚠️ **ВНИМАНИЕ**: Зафиксировать ВСЕ изменения, создать коммит и запушить на удалённый сервер. Использовать только когда уверены, что все изменения относятся к одной задаче.

## Рабочий процесс

### 1. Анализ изменений
Запустить параллельно:
- `git status` — показать изменённые/добавленные/удалённые/неотслеживаемые файлы
- `git diff --stat` — показать статистику изменений
- `git log -1 --oneline` — показать последний коммит для стиля сообщения

### 2. Проверки безопасности

**❌ ОСТАНОВИТЬСЯ и ПРЕДУПРЕДИТЬ при обнаружении:**
- Секреты: `.env*`, `*.key`, `*.pem`, `credentials.json`, `secrets.yaml`, `id_rsa`, `*.p12`, `*.pfx`, `*.cer`
- API-ключи: любые переменные `*_API_KEY`, `*_SECRET`, `*_TOKEN` с реальными значениями (не заглушки типа `your-api-key`, `xxx`, `placeholder`)
- Большие файлы: `>10MB` без Git LFS
- Артефакты сборки: `node_modules/`, `dist/`, `build/`, `__pycache__/`, `*.pyc`, `.venv/`
- Временные файлы: `.DS_Store`, `thumbs.db`, `*.swp`, `*.tmp`

**Валидация API-ключей:**
Проверить изменённые файлы на шаблоны типа:
```bash
OPENAI_API_KEY=sk-proj-xxxxx  # ❌ Обнаружен реальный ключ!
AWS_SECRET_KEY=AKIA...         # ❌ Обнаружен реальный ключ!
STRIPE_API_KEY=sk_live_...    # ❌ Обнаружен реальный ключ!

# ✅ Допустимые заглушки:
API_KEY=your-api-key-here
SECRET_KEY=placeholder
TOKEN=xxx
API_KEY=<your-key>
SECRET=${YOUR_SECRET}
```

**✅ Проверить:**
- `.gitignore` правильно настроен
- Нет конфликтов слияния
- Правильная ветка (предупредить при main/master)
- API-ключи только заглушки

### 3. Запрос подтверждения

Показать сводку:
```
📊 Сводка изменений:
- X файлов изменено, Y добавлено, Z удалено
- Итого: +AAA вставок, -BBB удалений

🔒 Безопасность: ✅ Нет секретов | ✅ Нет больших файлов | ⚠️ [предупреждения]
🌿 Ветка: [имя] → origin/[имя]

Будет выполнено: git add . → commit → push

Введите 'yes' для продолжения или 'no' для отмены.
```

**ЖДАТЬ явного "yes" перед выполнением.**

### 4. Выполнение (после подтверждения)

Запустить последовательно:
```bash
git add .
git status  # Проверить фиксацию
```

### 5. Генерация сообщения коммита

Проанализировать изменения и создать conventional commit:

**Формат:**
```
[тип]: Краткое описание (макс. 72 символа)

- Ключевое изменение 1
- Ключевое изменение 2
- Ключевое изменение 3
```

**Типы:** `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `build`, `ci`

**Пример:**
```
docs: Обновить README файлы концепций подробной документацией

- Добавить диаграммы архитектуры и таблицы
- Включить практические примеры
- Расширить разделы лучших практик
```

### 6. Коммит и пуш

```bash
git commit -m "$(cat <<'EOF'
[Сгенерированное сообщение коммита]
EOF
)"
git push  # При сбое: git pull --rebase && git push
git log -1 --oneline --decorate  # Проверить
```

### 7. Подтверждение успеха

```
✅ Успешно запушено на удалённый сервер!

Коммит: [хеш] [сообщение]
Ветка: [ветка] → origin/[ветка]
Изменено файлов: X (+вставки, -удаления)
```

## Обработка ошибок

- **git add не удался**: проверить разрешения, заблокированные файлы, инициализацию репозитория
- **git commit не удался**: исправить pre-commit хуки, проверить git config (user.name/email)
- **git push не удался**:
  - Non-fast-forward: `git pull --rebase && git push`
  - Нет удалённой ветки: `git push -u origin [ветка]`
  - Защищённая ветка: использовать PR-рабочий процесс

## Когда использовать

✅ **Хорошо:**
- Многофайловые обновления документации
- Функция с тестами и документацией
- Исправления ошибок в нескольких файлах
- Форматирование/рефакторинг всего проекта
- Изменения конфигурации

❌ **Избегать:**
- Не уверены, что фиксируется
- Содержит секреты/конфиденциальные данные
- Защищённые ветки без ревью
- Присутствуют конфликты слияния
- Нужна детализированная история коммитов
- Сбои pre-commit хуков

## Альтернативы

При необходимости контроля предложить:
1. **Выборочная фиксация**: просмотр/фиксация конкретных файлов
2. **Интерактивная фиксация**: `git add -p` для выбора патчей
3. **PR-рабочий процесс**: создать ветку → запушить → PR (использовать команду `/pr`)

**⚠️ Помните**: Всегда проверяйте изменения перед пушем. При сомнениях используйте отдельные git-команды для большего контроля.
