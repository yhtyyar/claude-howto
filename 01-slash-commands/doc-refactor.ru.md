# Рефакторинг документации

Реструктурируйте документацию проекта с учётом типа проекта:

1. **Анализировать проект**: определить тип (библиотека/API/веб-приложение/CLI/микросервисы), архитектуру и пользовательские персоны
2. **Централизовать документы**: перенести техническую документацию в `docs/` с правильными перекрёстными ссылками
3. **Корневой README.md**: оптимизировать как точку входа с обзором, быстрым стартом, сводкой модулей/компонентов, лицензией, контактами
4. **Документы компонентов**: добавить README-файлы уровня модуля/пакета/сервиса с инструкциями по настройке и тестированию
5. **Организовать `docs/`** по соответствующим категориям:
   - Архитектура, Справочник API, База данных, Дизайн, Устранение неполадок, Развёртывание, Вклад (адаптировать под нужды проекта)
6. **Создать руководства** (выбрать применимые):
   - Руководство пользователя: документация для конечных пользователей приложений
   - Документация API: эндпоинты, аутентификация, примеры для API
   - Руководство разработчика: настройка, тестирование, рабочий процесс вклада
   - Руководство по развёртыванию: production-деплой для сервисов/приложений
7. **Использовать Mermaid** для всех диаграмм (архитектура, потоки, схемы)

Документы должны быть краткими, лёгкими для сканирования и соответствующими типу проекта.
